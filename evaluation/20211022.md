# Evaluation: 2021-10-22 Analysis Result vs. Answer Key

- **Result:** `result/20211022.md`
- **Answer key:** `answers/2021-10-22_answer.txt` (SANS ISC October 2021 forensic contest)

## Overall Verdict

**Host identification (IP/hostname/username) is 3/3 correct, and the first-infection timestamp matches the answer key to the second.** However, **Qakbot — the malware family on 10.10.22.156 — was missed entirely** and replaced with generic "VNCStartServer / unidentified Trojans" labels. The result also repeats two recurring defects: a spurious lateral-movement narrative and a missed file hash. Malware family scoring: 1 fully correct, 1 partially correct, 1 missed.

## Wrong (Most Important)

**Qakbot missed on 10.10.22.156.** The answer key identifies three HTTP downloads of a Qakbot DLL (URLs ending `/44491/6090605324.dat`) with post-infection traffic including TCP 23.111.114.52:65400 and encrypted SMTP to multiple mail servers. The result instead labeled this host "VNCStartServer (Backdoor/RAT) and unidentified Trojans." The observed VNC C2 (45.153.241.142) and SMTP-capable behavior are in fact well-known **Qakbot modules** — the pipeline saw Qakbot's components but failed to attribute them to the family. This is the recurring "weak host-to-malware mapping" defect.

**Spurious lateral-movement narrative, again.** The result frames 10.10.22.156 as a "Compromised Pivot" conducting svcctl/RPC lateral movement against .157, .158, and the DC (10.10.22.22). The answer key describes **three independent email-borne infections** with no lateral-movement chain. While Qakbot does probe SMB/RPC as part of its spreading module, presenting this as the incident's pivot narrative — and listing the domain controller as a target — repeats the "infrastructure-as-victim" error class from the 2021-06-16 evaluation.

## Missing

- **Qakbot delivery URLs:** three HTTP requests ending in `/44491/6090605324.dat` — absent
- **Qakbot DLL SHA256:** `95928b331e9942e4709b41ec8b59eb6f9e068f53ef2eeb15009feafd4ff5138d` — result reports hashes "unknown"; this is the second consecutive case where an extractable file hash was missed (Zeek files.log extraction gap)
- **Qakbot post-infection indicator:** TCP to `23.111.114.52:65400`
- **Encrypted SMTP burst** from 10.10.22.156 (a strong Qakbot signal) — not reported
- **Initial infection vectors:** the malicious emails (Order.7z / NanoCore, firebase link / BazarLoader zip, Document-1975072354.zip / Qakbot xlsm). *Mitigating factor: the .eml files were separate contest inputs, not part of the pcap, so the pipeline could not have seen them — but the report also never states the delivery vector is unknown.*

## Unverified Claims in the Result

- `172.67.139.101`, `178.23.190.8`, `190.14.37.244`, `194.36.191.35` as "malicious EXE delivery" — not listed in the answer key; plausibly the Qakbot `.dat` URL hosts, but the "EXE" characterization conflicts with the answer key's DLL description
- "Dridex identified via JA3 hash" on .158 — JA3 collisions between Dridex and BazarLoader TLS stacks are common; the answer key confirms BazarLoader only

## Summary

Identity mapping is perfect and timestamps are the most accurate of the evaluated runs (patient-zero time exact to the second). The dominant defect is the failure to name Qakbot despite observing its component behaviors (VNC module, SMTP spam capability, port-65400 C2), compounded by the recurring spurious lateral-movement/pivot narrative and a second consecutive missed file hash. Priority fixes: (1) family attribution layer that maps module-level behaviors to known families, (2) suppress lateral-movement claims when SMB/RPC traffic targets the domain controller in an AD environment, (3) hash extraction from Zeek files.log.

## Addendum: Re-scoring Under the Pcap-Only Criterion

The answer key used inputs the pipeline never had (the .eml files, any.run/Tria.ge sandbox runs, VirusTotal). Re-scored against what is achievable from **pcap + Suricata/Zeek + offline LLM only**, verified against the actual `output/20211022/` artifacts:

**Qakbot miss → reclassified as an attribution ceiling, not an LLM error.** The eve.json contains **zero Qakbot/Qbot-named signatures**. The only family-named alerts that fired were: `ET MALWARE NanoCore RAT CnC 7`, `ET JA3 Hash - Possible Dridex`, `ET MALWARE VNCStartServer USR/BOT Variant CnC Beacon`, and `ET HUNTING ... EmerDNS TLD (.bazar)`. The result's labels (NanoCore ✓, "Dridex via JA3," "VNCStartServer," "Bazar") are a faithful transcription of the maximum family attribution the evidence permitted. Naming Qakbot required sandbox/VT lookups the offline pipeline cannot perform.

**Hash miss → reclassified as an LLM synthesis failure (worse than previously assessed).** The Qakbot DLL SHA256 `95928b...138d` **is present in `evidence.json` `files[]`** (mime `application/x-dosexec`, 911,360 bytes, count 3 — the three .dat downloads). Extraction worked; the LLM had the hash in its evidence bundle and still wrote "hashes unknown." This is fully penalizable under the pcap-only criterion and is a prompt/synthesis bug, not a Zeek gap.

**Lateral-movement narrative → root cause partially located in the evidence schema.** `evidence.json` has a top-level section literally named `lateral_movement` containing normal AD traffic: SMB pairs toward the DC, DCERPC endpoints `drsuapi/samr/netlogon/lsarpc/svcctl`, Kerberos, LDAP. The bundle's own naming primes the LLM to narrate lateral movement. Fix belongs in the evidence builder (rename to `internal_ad_traffic` + annotate infra roles), not only in the LLM prompt.

**Pcap-only verdict:** extraction layer performed essentially perfectly; the LLM reached the signature-attribution ceiling on families; the remaining true defects are (a) ignoring the hash present in its own evidence, and (b) the schema-induced lateral-movement story.
