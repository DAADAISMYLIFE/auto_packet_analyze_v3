## Wrong (Most Important)

**Infection scenario misjudgment.** The result describes a single kill chain: "10.6.15.119 was the initial compromise, then spread to .93 and .187 via SMB/NTLM lateral movement." The answer key shows **three independent infections** from separate malspam campaigns:

- .119: Google Feedproxy link → malicious Word doc via `ststephenskisugu.church` (14:35 UTC)
- .187: malicious zip via `solarwindsonline.com` (15:30 UTC)
- .93: AgentTesla, separate infection (15:44 UTC)

The SMB/NTLM traffic interpreted as "lateral movement" is most likely normal Active Directory authentication against the domain controller (10.6.15.5). Framing the DC as an attack target is the same error class as the recurring "infrastructure-as-victim" issue.

There is also an internal contradiction in the timeline: Cobalt Strike beaconing at 14:33:54 is placed *before* the payload download at 14:37, which is causally impossible for the described chain.

## Summary

Victim identification and malware classification are at answer-key level. The single biggest defect is collapsing three independent infections into one lateral-movement chain — a patient-zero/scenario error, likely caused by misreading normal AD authentication as attacker activity. Secondary defects: incomplete Qakbot IOC coverage and a missed file hash.

## Addendum: Re-scoring Under the Pcap-Only Criterion

The answer key drew on external material (Tria.ge/URLhaus for the Qakbot sample's macro-generated URLs, the author's own TA551 research). Re-scored against what is achievable from **pcap + Suricata/Zeek + offline LLM only**, verified against `output/20210616/` artifacts:

**Converted to known limits (not LLM errors):**
- The full `khangland.pro` / `jaipurbynite.com` URL paths came from sandbox analysis of the Excel macro — only the domains/SNI are pcap-visible.
- The Qakbot zip itself was unrecoverable from the pcap even for the answer-key author (packet loss).

**Remain valid errors — the evidence was in the bundle and the LLM omitted it:**
- `solarwindsonline.com` (Qakbot zip delivery) and `162.244.83.95` (second Cobalt Strike server) are **present in `evidence.json`** but absent from the report.
- The Ficker Stealer EXE SHA256 `dee4bb7d...c2019` is **present in `evidence.json`** — the report claimed "no hashes identified." Extraction worked; synthesis dropped it.
- The independent-infections-merged-into-one-chain scenario error stands: the three separate delivery flows are all pcap-visible, and the "lateral movement" is normal AD authentication packaged under the evidence bundle's misleading `lateral_movement` section name (schema-induced, same as the other cases).

**Pcap-only verdict:** family attribution (AgentTesla, Hancitor+CS+Ficker, Qbot) was fully achievable from ET signatures and was achieved. The true defects are evidence-utilization failures (in-bundle IOCs and hash omitted) and the schema-induced lateral-movement narrative.
