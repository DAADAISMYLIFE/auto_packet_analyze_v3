## Wrong

- **10.6.13.3 misframed as a compromise target.** The result lists it as "Secondary Target — internal target of lateral movement (SMB/RPC)." The answer key identifies 10.6.13.3 as the **DHCP server**; observed SMB/RPC/LDAP toward it is routine Windows infrastructure traffic. This is the **third consecutive case** of the infrastructure-as-victim / spurious-lateral-movement error (2021-06-16: DC as NTLM target; 2021-10-22: DC as svcctl pivot target; here: DHCP server).
- **Corrupted IP in IOC list:** the result prints `67.2int.228.199` — a malformed token (the timeline section has the presumably correct `67.217.228.199`). An IOC destined for a block policy must be machine-valid; this would break automated policy application, which is the project's end goal.
- **Campaign label divergence:** result says "LandUpdate808 Exploit Kit" (likely from an ET signature name); the answer key attributes the chain to **KongTuke TDS** delivering **Pupy RAT**. LandUpdate808 and KongTuke are closely related fake-update/TDS campaign names, so this is partial rather than flat wrong — but the result names no final payload family at all, continuing the family-attribution gap from 2021-10-22.

## Missing

- **`truglomedspa.com` — the true origin of the chain** (15:34:45): a legitimate compromised spa website where the KongTuke script was injected. The result's timeline starts one step later at hillcoweb.com, so the report cannot answer "how was the victim lured?"
- **`dng-microsoftds.com` and the `/zh0GPFZdKt` obfuscated PowerShell delivery** (15:35:38): the actual script-delivery step. The result mentions PowerShell C2 activity but not the domain/URL that served the script.
- **C2 POST behavior detail:** answer key notes repeated POSTs with `application/octet-stream` bodies to the three C2 domains — useful for a block/detection pattern; result describes C2 generically.
- **C2 IP mismatch (unresolved):** answer key gives 104.21.16.1 (`windows-msgas.com`) and 104.16.230.132 (trycloudflare); result gives 104.21.112.1 and 104.21.24.186. All are Cloudflare ranges and DNS answers rotate, so both may have been true at capture time — but the discrepancy was not reconcilable from the two documents.

## Unverified Claims in the Result

- "LandUpdate808 Exploit Kit" as the infection mechanism — the chain is a fake-captcha/TDS redirect, not a classic exploit kit; the "EK" framing overstates what the traffic shows.
- Timeline entry "15:35:51 PowerShell execution" — plausible sequencing next to the answer key's 15:35:38 script delivery, but the result does not say what evidence distinguishes PowerShell specifically.

## Summary

Best victim-identification performance so far (all four identity fields exact, first-activity timestamp within seconds) and good C2 domain coverage. The report fails on chain completeness — missing the compromised entry-point site and the script-delivery domain, i.e. the two steps that explain *how* the infection happened — and repeats the infrastructure-as-victim error, now with a DHCP server. New defect class: a corrupted IOC string (`67.2int.228.199`) that would break downstream automated blocking. Cross-case priority fixes remain: (1) AD/infra role detection (DC, DHCP, DNS servers) before claiming lateral movement, (2) payload family attribution, (3) IOC output validation (regex-check IPs/domains before report emission).

The answer key relied heavily on external lookups (VirusTotal, urlscan.io, ThreatFox, third-party blog analyses) for the KongTuke/Pupy attribution. Re-scored against **pcap + Suricata/Zeek + offline LLM only**, verified against `output/20250613/` artifacts:

- **KongTuke/Pupy naming was unattainable.** The only campaign-named signatures in eve.json are `ET EXPLOIT_KIT LandUpdate808 Domain in TLS SNI/DNS Lookup (hillcoweb.com)`. - The `truglomedspa.com → hillcoweb.com` KongTuke-injection explanation (compromised legitimate site) is OSINT-derived; from the pcap alone one can only observe the referral sequence.

**Remain valid errors — the evidence was in the bundle and the LLM omitted or corrupted it:**
- `truglomedspa.com` and `dng-microsoftds.com` are **both present in `evidence.json`** external domains. A time-ordered review of pre-alert traffic would have surfaced the full chain; the report starts one hop late. Patient-zero-precursor omission stands.
- The corrupted IOC `67.2int.228.199` stands — a generation-level defect independent of evidence quality.
- The DHCP server (10.6.13.3) framed as a lateral-movement target stands, again primed by the evidence bundle's `lateral_movement` section naming.

**Pcap-only verdict:** attribution reached its offline ceiling (LandUpdate808). Remaining true defects are evidence-utilization (in-bundle precursor domains omitted), IOC string corruption, and the schema-induced lateral-movement framing.
